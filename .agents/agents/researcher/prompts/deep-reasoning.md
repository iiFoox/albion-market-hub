## Technique 1: 5 Whys Root Cause Analysis

For complex bugs or architectural problems, drill down to the actual root cause.

```
PROTOCOL:
1. STATE the problem clearly and specifically
2. Ask WHY #1 — Why is this happening? → Answer with evidence
3. Ask WHY #2 — Why does THAT happen? → Answer with evidence
4. Ask WHY #3 — Why does THAT happen? → Answer with evidence
5. Ask WHY #4 — Why does THAT happen? → Answer with evidence
6. Ask WHY #5 — Why does THAT happen? → Answer = ROOT CAUSE
7. VERIFY — Is the root cause actionable? Can we fix it?

RULES:
- Each "why" must be answered with evidence, not speculation
- If you can't answer a "why" with evidence, mark it as UNKNOWN
- The root cause should be something we can actually change
- Don't stop early — superficial causes lead to superficial fixes
- Don't go past 5 unless the root cause isn't clear yet
```

### 5 Whys Example

**Problem:** "API response times increased from 200ms to 2000ms this week."

```
WHY #1: Why are API responses slow?
→ Database queries are taking 10x longer than last week.
  EVIDENCE: Query monitoring shows avg query time went from 50ms to 500ms.

WHY #2: Why are database queries slower?
→ The query planner is doing sequential scans instead of index scans on the orders table.
  EVIDENCE: EXPLAIN ANALYZE shows Seq Scan on orders (2.3M rows).

WHY #3: Why is the query planner choosing sequential scans?
→ The index on orders.created_at was invalidated after a recent migration.
  EVIDENCE: pg_stat_user_indexes shows the index has 0 scans since Tuesday.
  Migration on Monday altered the column type from timestamp to timestamptz.

WHY #4: Why did the migration invalidate the index?
→ ALTER COLUMN TYPE drops and does NOT recreate dependent indexes in PostgreSQL.
  EVIDENCE: PostgreSQL docs confirm this behavior. The migration script did not
  include a CREATE INDEX statement after the ALTER.

WHY #5: Why didn't the migration script recreate the index?
→ ROOT CAUSE: The migration was auto-generated by Prisma and the developer didn't
  review the generated SQL. Prisma's migrate doesn't always preserve custom indexes
  during type changes.

ACTION: (1) Recreate the index immediately. (2) Add a post-migration index
verification step to the deployment pipeline. (3) Store this as a learning:
"Always review Prisma-generated migrations for index preservation."
```

## Technique 2: Failure Mode Analysis (FMEA)

Systematically analyze how a system or feature can fail.

```
FOR EACH COMPONENT/FEATURE:

1. IDENTIFY all possible failure modes
   → What can go wrong? (input, processing, output, timing, resources)

2. ASSESS each failure mode:
   → Severity (1-10): How bad is it if this happens?
   → Occurrence (1-10): How likely is it to happen?
   → Detection (1-10): How hard is it to detect BEFORE it reaches users?
   → RPN = Severity × Occurrence × Detection

3. PRIORITIZE by RPN score
   → RPN > 200: Critical — must address immediately
   → RPN 100-200: High — should address before release
   → RPN 50-100: Medium — address when possible
   → RPN < 50: Low — monitor

4. DEFINE mitigation for high-RPN items
```

### FMEA Example

**Feature:** "Payment processing via Stripe"

| Failure Mode | Severity | Occurrence | Detection | RPN | Mitigation |
|---|---|---|---|---|---|
| Stripe API timeout | 8 | 4 | 3 | 96 | Retry with backoff, idempotency key |
| Duplicate charge | 10 | 3 | 5 | 150 | Idempotency key mandatory |
| Webhook lost | 8 | 3 | 7 | 168 | Webhook retry + reconciliation job |
| Invalid card (user error) | 3 | 7 | 1 | 21 | Client validation + clear error msg |
| Partial refund fails | 6 | 2 | 6 | 72 | Admin dashboard + manual refund option |
| Currency mismatch | 9 | 2 | 8 | 144 | Server-side currency validation |
| Stripe key exposed | 10 | 1 | 9 | 90 | Env vars only, key rotation alerts |

**Priority:** Webhook lost (168) > Duplicate charge (150) > Currency mismatch (144)

## Technique 3: Trade-Off Quadrant Analysis

When evaluating solutions, plot options on a 2×2 quadrant to make trade-offs visible.

```
CHOOSE TWO COMPETING DIMENSIONS:
Axis X: [Dimension A] (e.g., Implementation Speed)
Axis Y: [Dimension B] (e.g., Scalability)

PLOT OPTIONS:
┌─────────────────────────────────────┐
│  High B            │  High B        │
│  Low A             │  High A        │
│                    │                │
│  (Good long-term,  │  (IDEAL but    │
│   slow to build)   │   does it      │
│                    │   exist?)      │
├────────────────────┼────────────────┤
│  Low B             │  Low B         │
│  Low A             │  High A        │
│                    │                │
│  (AVOID)           │  (Quick win,   │
│                    │   tech debt)   │
└─────────────────────────────────────┘

DECISION: Which quadrant is acceptable for THIS project?
→ Startup MVP: bottom-right acceptable (speed > scalability)
→ Enterprise: top-left acceptable (scalability > speed)
→ Mature product: top-right required (both or nothing)
```

## Technique 4: Adversarial Analysis

Think like an attacker or malicious user to find weaknesses.

```
PERSONAS:
1. MALICIOUS USER: What can they abuse?
   → Can they access data they shouldn't?
   → Can they cause denial of service?
   → Can they manipulate prices, permissions, or state?

2. INCOMPETENT USER: What can they break accidentally?
   → What happens if they double-click submit?
   → What happens with empty/null/very long inputs?
   → What happens if they use the back button at the wrong time?

3. EDGE CASE EXPLORER: What about unusual scenarios?
   → Timezone boundary conditions
   → Unicode/emoji in text fields
   → Concurrent operations by multiple users
   → Network interruption mid-operation

4. FUTURE DEVELOPER: What will they misunderstand?
   → Are there implicit assumptions not documented?
   → Are there "magic numbers" or unclear constants?
   → Is the error handling adequate for debugging?
```

## When to Use Deep Reasoning

| Trigger | Technique |
|---|---|
| Complex bug that resists standard debugging | 5 Whys |
| Evaluating reliability of a critical feature | FMEA |
| Comparing solution approaches with trade-offs | Trade-Off Quadrant |
| Security review or robustness assessment | Adversarial Analysis |
| Performance regression investigation | 5 Whys + FMEA |
| Architecture decision with long-term impact | Trade-Off Quadrant + FMEA |
